# Workflow: /review-code

> Pre-PR code review checklist. Detects standards deviations, potential bugs, and security issues.
> This is a quality and compliance validation — not a refactor.

---

## When to Use This Workflow

- Before opening a pull request to the main branch
- When validating code generated by an agent
- When reviewing a teammate's code
- As a self-review at the end of a feature

---

## Required Information

1. **What is being reviewed?** — specific module, PR, set of files
2. **Layer(s):** Backend / Frontend / Mobile / Multiple
3. **Special focus (if any):** security / performance / UX / database / accessibility

---

## Backend Checklist

### Entities / Data Models
- [ ] Table/collection name follows project naming convention (e.g. `snake_case` plural)
- [ ] Primary key uses the project's standard ID type (UUID, CUID, auto-increment)
- [ ] Column/field names follow the naming convention consistently
- [ ] Foreign keys are explicitly declared with the correct reference
- [ ] `createdAt` and `updatedAt` timestamps are present
- [ ] Sensitive fields (passwords, tokens, secrets) are excluded from default selects and serialization
- [ ] All public fields have API documentation annotations (`@ApiProperty` or equivalent)

### DTOs / Request Schemas
- [ ] All fields have explicit type annotations
- [ ] All fields have validation rules (required/optional, min/max, format)
- [ ] Update/patch schemas correctly mark all fields optional
- [ ] Pagination/list query schemas include `page`, `limit`, and sort parameters with defaults
- [ ] No extra fields accepted beyond what is explicitly defined (strict mode / `whitelist`)

### Service Layer
- [ ] Uses structured logging — no `console.log` in production code
- [ ] All config/env values accessed through a config service — no `process.env` direct reads
- [ ] Correct HTTP exception types thrown (`NotFoundException`, `ConflictException`, `ForbiddenException`, etc.)
- [ ] Audit/event logging present on all write operations (CREATE, UPDATE, DELETE)
- [ ] Zero presentation logic — the service does not build HTML, navigate, or format for display
- [ ] No use of `any` type — all types are explicit
- [ ] Async operations correctly awaited — no dangling promises

### Controller / Route Handler
- [ ] Routes follow the project URL convention (kebab-case, versioned prefix, plural resources)
- [ ] Each endpoint has documentation annotation (`@ApiOperation`, `@summary`, or equivalent)
- [ ] Auth guards and role/permission guards applied correctly
- [ ] ID path parameters validated and typed (UUID pipe or equivalent)
- [ ] Request body and query parameters validated through schema/DTO
- [ ] Zero business logic in the controller — it only delegates to the service
- [ ] Response shape consistent with the rest of the API

### Security
- [ ] No endpoint is publicly accessible without an explicit, intentional decision
- [ ] All queries are scoped to the correct owner/tenant/organization
- [ ] The acting user's identity comes from the verified auth token, not from the request body
- [ ] Rate limiting applied to sensitive endpoints (auth, password reset, public submissions)
- [ ] No sensitive data (passwords, tokens, PII) returned in responses or logged

### Database
- [ ] A migration exists for every schema change — no reliance on auto-sync in production
- [ ] Migrations reviewed: no accidental column drops, no data loss on existing rows
- [ ] New permission/role entries added to the seed file if applicable
- [ ] Indexes added for columns used in frequent queries or unique constraints

## Frontend Checklist

### Component Architecture
- [ ] Component is self-contained (standalone/isolated, no implicit global dependencies)
- [ ] Dependency injection uses the project's standard pattern
- [ ] Props/inputs and events/outputs use the modern API (`input()`, `output()`, or framework equivalent)
- [ ] Template control flow uses modern syntax (`@if`/`@for` or framework equivalent — not legacy directives)
- [ ] Template is inline or in a co-located file — not referencing a distant template
- [ ] Template-accessible members are protected/internal; purely internal logic is private

### Reactive State
- [ ] Mutable state uses signals or the equivalent reactive primitive — no legacy observables for local state
- [ ] Derived/computed values use `computed()` or equivalent — not recalculated on every render
- [ ] Side effects (data fetching, DOM interaction) are properly encapsulated with the correct reactive effect API
- [ ] No manual subscriptions that need to be manually cleaned up — or, if unavoidable, properly disposed

### HTTP / Data Fetching
- [ ] HTTP calls go through the project's HTTP abstraction (no raw HTTP client directly in components)
- [ ] All async operations use `async/await` consistently
- [ ] Loading state tracked and exposed to the template
- [ ] Errors caught and surfaced to the user via the standard notification/toast mechanism
- [ ] No HTTP calls in `ngOnInit` / `useEffect` that lack proper cleanup on component destruction

### Styles
- [ ] Colors use CSS custom properties / design tokens — no hardcoded hex values
- [ ] Dark mode compatible (styles work under the `.dark` class or `prefers-color-scheme`)
- [ ] Responsive design: mobile breakpoints handled
- [ ] Shared/common components used where available instead of duplicating markup

### UX
- [ ] Loading state is visually indicated (skeleton, spinner, or progress indicator)
- [ ] Empty state is handled (not a blank area — show a message or empty-state component)
- [ ] Success feedback shown after write operations (toast, confirmation message)
- [ ] Destructive actions (delete, overwrite) require explicit user confirmation
- [ ] UI elements that require permissions are hidden or disabled for unauthorized users

### Accessibility
- [ ] Interactive elements are keyboard-accessible
- [ ] Form inputs have associated labels
- [ ] Images have meaningful `alt` text (or `alt=""` if decorative)
- [ ] Color alone is not the only visual indicator of state

---

## Mobile Checklist

- [ ] UI components imported individually (not as a monolithic module import)
- [ ] Icons registered explicitly before use
- [ ] Routes organized under the correct role namespace
- [ ] Role guard applied on restricted routes
- [ ] Pull-to-refresh implemented on list screens
- [ ] Platform-specific behavior (camera, file access, push notifications) handled with proper permission requests
- [ ] All Angular / frontend checklist items above also apply

## Review Report Format

```
## Code Review — [module/feature] — [date]

### Summary
[1-2 sentences on overall state: approved / requires changes / blocked]

### What is Correct
- [thing done well 1]
- [thing done well 2]

### Required Changes

#### [File: path/to/file.ts]
- **Line [N]:** [problem] → [suggested fix]
- **Line [N]:** [problem] → [suggested fix]

### Blockers (do not merge without resolving)
- [Critical issue affecting security, data integrity, or core functionality]

### Verdict
[ ] Approved — ready to merge
[ ] Approved with minor changes — no re-review needed
[ ] Changes required — needs another review pass
```

## Verification Commands

Run these before submitting a PR or marking a review complete.

```bash
# Build check — catches type errors and compile failures
<your build command>

# Test suite
<your test command>

# Linting
<your lint command>

# Search for common anti-patterns
grep -r "console\.log" src/ --include="*.ts"
grep -r ": any" src/ --include="*.ts"
grep -r "TODO\|FIXME\|HACK" src/ --include="*.ts"
```

Adapt the commands to your project's tooling (Nx, Turbo, plain npm scripts, etc.).
